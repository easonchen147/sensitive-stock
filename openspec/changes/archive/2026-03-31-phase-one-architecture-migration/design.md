## Context

当前项目是以 `app.py` 为入口的 Streamlit 单体应用。页面跳转、状态管理、业务服务、第三方数据访问和 AI 调用都堆叠在同一个运行时中，导致几个现实问题：

- 页面层和业务层边界模糊，`pages/backtest_page.py`、`market_data.py` 等文件持续膨胀。
- 任何一个外部依赖异常都可能直接中断整个 UI 交互过程。
- 现有能力难以以 API 的形式复用，也不利于后续接入更强的前端体验或测试体系。
- 文档、配置和代码的实际状态已经出现明显偏差，必须在迁移过程中同步建立新的工程基线。

第一阶段迁移的目标不是“一次性把所有 Streamlit 逻辑彻底替换完”，而是建立一个可运行、可扩展、可继续迁移的前后端分离骨架。优先级最高的是迁通一条关键主链路，并对其余能力留出清晰的服务边界和占位实现。

## Goals / Non-Goals

**Goals:**

- 建立 `backend/` Flask 后端工作区，采用应用工厂、蓝图路由、服务层与 schema 层组织方式。
- 建立 `frontend/` Next.js + React 前端工作区，提供统一布局、导航、API client 与功能页骨架。
- 优先迁移股票回测主链路，使其可以通过 HTTP API 从前端提交参数并返回结果。
- 为条件选股、行情中心、AI 诊股、因子分析、组合优化预留明确的后端 API 入口与前端页面壳层。
- 引入 `uv + Poetry` 的 Python 工作流说明和工程配置，替代当前仅依赖 `requirements.txt` 的方式。
- 更新 README、迁移说明和目录结构文档，使开发者以新架构为默认认知模型。

**Non-Goals:**

- 第一阶段不要求删除现有 Streamlit 代码，也不要求一次性迁移所有交互细节。
- 第一阶段不重写现有量化核心算法，只对现有模块做适配和封装。
- 第一阶段不引入数据库、任务队列、认证体系或生产部署编排。
- 第一阶段不承诺所有外部行情/新闻接口在新架构中都已经完全恢复到生产可用状态。

## Decisions

### 1. 后端选用 Flask 而不是继续保留 Streamlit 入口

选择 Flask 的原因是它足够轻量，适合第一阶段快速建立 REST API 边界，同时能用最少改动承接现有 Python 业务模块。相比直接把现有逻辑塞进 FastAPI 或更重的后端框架，Flask 的接入阻力更低，也更贴合“先搭骨架、先迁主链路”的节奏。

备选方案：

- 继续用 Streamlit 暴露页面，延后 API 化：无法解决单体耦合问题，不满足迁移目标。
- 直接切到 FastAPI：类型体验更好，但第一阶段收益不足以覆盖额外的迁移与学习成本。

### 2. 第一阶段通过“适配封装”复用现有量化模块，而不是立即移动全部文件

后端服务层将优先通过 `backend/app/services` 对现有 `backtesting/*`、`factor_analysis.py`、`portfolio_optimizer.py` 等模块做薄适配。这样可以快速打通主链路，同时把未来重构点集中到新服务层边界上，避免一开始就进行大规模移动文件。

备选方案：

- 立即把所有根目录模块迁入 `backend/`：改动面过大，容易引入回归，而且会和第一阶段“保留现有功能边界”的要求冲突。

### 3. 已迁通主链路选定为“回测页面 → 回测 API → 旧回测引擎”

回测链路已有相对清晰的流程编排：`BacktestRequest` → `BacktestService` → 数据获取 → 策略执行 → 回测引擎。相比条件选股、市场资讯、AI 诊股，它对外部接口形态依赖更少，最适合作为第一阶段的示范链路。

备选方案：

- 先迁条件选股：高度依赖东方财富接口与指纹配置，不适合作为第一条示范链路。
- 先迁资讯/板块：依赖多个不稳定数据源，验证成本高。

### 4. 前端选用 Next.js App Router + React 最新稳定版

前端采用 Next.js App Router 作为页面壳层，使用 React 最新稳定版本，并通过 Server/Client Component 组合实现统一导航、页面路由和 API 访问层。第一阶段只实现一个能工作的应用壳和回测页面，其余页面保留功能说明和迁移占位。

备选方案：

- 用 Vite + React：可以更轻，但缺少 Next.js 在未来页面扩展、部署和路由组织上的优势。
- 直接迁成复杂设计系统：不符合第一阶段目标，容易把注意力从架构迁移偏向视觉细节。

### 5. Python 依赖管理体现为“Poetry 定义项目，UV 负责环境与执行”

后端工作区使用 `pyproject.toml`（Poetry）作为依赖与项目元数据载体，同时在文档中约定使用 UV 创建虚拟环境和执行命令。这样既满足新工作流要求，也为未来统一锁定依赖和自动化脚本留下空间。

备选方案：

- 仅保留 `requirements.txt`：无法体现迁移后的工程工作流。
- 完全切到单一工具并删除 Poetry：不符合本次变更要求。

### 6. 同步修复已知的数据提供器接口错位

`factor_analysis.py` 和 `portfolio_optimizer.py` 当前调用了不存在的 `SmartDataProvider.get_data()`。第一阶段如果要为这两类能力预留后端入口，就必须先把它们改为使用统一的 `HistoricalDataRequest` / `get_ohlcv()` 契约。

备选方案：

- 保持现状，仅留 TODO：会导致新后端占位接口一接通就报错，骨架不可用。

## Risks / Trade-offs

- [前后端骨架先行，功能迁移未完全覆盖] → 在 README 和迁移状态文档中明确“已迁移 / 未迁移”边界，避免误解。
- [现有外部数据源不稳定，回测 API 仍可能受网络或上游接口影响] → 保留清晰错误模型，前端显示可操作的失败信息。
- [前端与旧业务模块并行存在，仓库结构临时变复杂] → 用新目录结构文档说明职责，后续阶段再逐步清理旧入口。
- [本地环境未必具备 UV] → 文档中给出推荐工作流，并说明本轮验证范围以 Poetry、pytest、npm build 为主。
- [Next.js / React 版本选择若过旧会削弱“新基线”意义] → 以官方最新稳定版为准，并在文档中注明选型依据。

## Migration Plan

1. 补齐 OpenSpec 的设计、规格和任务制品，解除变更阻塞状态。
2. 创建 `backend/` Flask 工程，加入健康检查、能力清单和回测 API。
3. 为回测结果建立标准 JSON schema，并封装旧回测服务的输出序列化。
4. 修复因子分析和组合优化的数据接口错位，确保后续 API 占位可复用。
5. 创建 `frontend/` Next.js 工程，加入首页、导航、回测页和能力占位页。
6. 打通前端回测页面到后端回测 API 的联调主链路。
7. 更新 README、迁移文档、目录结构和启动方式，新增 MIT LICENSE。
8. 通过测试、构建和语法检查验证阶段性结果，并更新 `tasks.md` 进度。

回滚策略：

- 新架构以新增目录为主，不直接删除旧 Streamlit 入口；若新链路不可用，仍可回退到旧代码路径继续修复。
- 回测 API 服务层保持对现有量化模块的薄适配，若新序列化层出现问题，可以单独回滚 backend 变更，不影响底层回测实现。

## Open Questions

- 条件选股接口所需的东方财富指纹后续将如何安全迁入后端配置体系，是否需要单独的安全配置改造变更。
- AI 调用是否要在第二阶段统一封装成 provider 层，而不是继续散落在多个旧模块中。
- 第二阶段是否保留旧 Streamlit 入口作为临时运维工具，还是在功能迁移完成后彻底移除。
