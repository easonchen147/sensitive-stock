## Context

当前前端已经脱离旧 Streamlit 入口，但页面成熟度分层明显：

- 首页、回测页、市场页已有一定工作台形态；
- `screener`、`diagnosis` 仍是 placeholder；
- `factors`、`portfolio` 尚未正式进入前端；
- 数据接入还不是严格 OpenAPI-first。

用户这次不仅要求“全部页面全面样式 UI 设计重构”，还明确指定使用 `huashu-design`。这意味着本设计不能停留在常规“组件库美化”层面，而要先回答：**这套系统适合哪一种研究工作台视觉语言**。

按照 Huashu 的方法，本轮在没有既定设计系统和 Figma 的前提下，先做方向判断，再落统一页面语法。

## Goals / Non-Goals

**Goals:**

- 建立适合股票研究、回测、情报分析的统一页面语言。
- 让所有主页面都进入同一套布局、状态表达和交互语法。
- 以 `openapi.json` 为前端数据契约源，而不是继续扩大手写 helper 漂移。
- 让设计验证可重复，通过截图/状态验证约束真实交付质量。

**Non-Goals:**

- 本轮不把前端做成营销站、品牌宣传页或装饰性 hero 网站。
- 本轮不为每个页面发明独立视觉风格；目标是统一系统语言。
- 本轮不在该 change 内定义后端业务本身的完成条件，那由相邻提案承接。

## Decisions

### 1. 采用 Huashu 风格顾问模式中的“Quiet Capital Terminal”方向

在本系统里，最合适的不是插画式、品牌秀场式或极端实验风格，而是偏：

- 研究终端
- 操作台
- 数据密集但有秩序
- 强状态语言

因此本 change 选择的主方向是 **Quiet Capital Terminal**：

- 低噪声底色
- 强层级对比
- 面向扫描和决策的布局
- 用结果、状态和数据关系建立美感，而不是靠装饰性视觉堆砌

备选方向：

- `Research Notebook`：更偏叙述和注释，不适合作为全站主风格。
- `Signal Radar Board`：更偏监控面板，适合局部页面但不适合作为所有页面主语法。

### 2. 建立统一页面模板，而不是页面级各写各的

所有研究工作台统一采用以下大框架：

- 顶部摘要带：页面目标、状态、关键指标
- 控制区：筛选、参数、时间范围、执行动作
- 主结果区：表格、图表、报告、卡片化数据块
- 解释侧栏：假设、来源、风险、降级说明

这样可以让用户在不同页面之间迁移时，学习成本最小化。

### 3. OpenAPI 驱动的类型与 API client 成为前端正式事实来源

本轮不继续放大手写 `fetch + type` 漂移，而是要求：

- 以 backend 发布的 `openapi.json` 为准；
- 生成或对齐 `frontend/types/*`；
- 统一 `frontend/lib/*` 的 client 行为；
- 页面只消费稳定 contract。

### 4. 全站状态语言统一，placeholder 被正式工作台替换

页面完成后，系统应只保留以下正式状态：

- loading
- empty
- degraded
- error
- ready

而不是继续通过 placeholder brief 承担真实页面职责。

### 5. 设计交付必须可验证

按照 Huashu 的要求，本轮前端改版不能只停留在“代码看起来写了”，而应至少通过：

- 页面截图检查
- 关键页面状态验证
- 主要工作台布局一致性复核

## Risks / Trade-offs

- `[视觉重构范围大]` -> 通过统一页面模板与共享状态组件减少重复设计成本。
- `[OpenAPI 接入会暴露既有接口漂移]` -> 先由后端 OpenAPI 提案稳定契约，再切前端 client。
- `[过度设计导致操作效率下降]` -> 以扫描效率和结果解释为第一原则，避免营销站式构图。
- `[单页风格过强不适合全站]` -> 选用 Quiet Capital Terminal 这种低噪声、高适配性的主方向。

## Migration Plan

1. 明确全局设计方向、页面模板和状态组件语法。
2. 统一登录页、首页、导航壳层与共享布局。
3. 逐页重构回测、行情、选股、诊股、因子、组合工作台。
4. 以 OpenAPI 为准接通所有数据接口和类型。
5. 用截图和交互验证确认全站一致性。

回滚策略：

- 页面级模板和共享组件可以逐页替换，不必一次性全量切换。
- 若某一工作台改版风险较高，可先保留旧页面并并行验证新模板。
