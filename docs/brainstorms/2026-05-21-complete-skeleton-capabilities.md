# 完成四类半成品能力 Brainstorm

## 背景

当前系统已经形成可用的登录、回测、行情工作台主链路，但仍有四类能力停留在半成品状态：

1. `screener` 条件选股：仅保留壳层入口与 placeholder API。
2. `diagnosis` AI 诊股：仅保留入口与状态说明，未形成真实诊断链。
3. `factors` 因子分析：底层模块存在，但没有服务层、API 与前端工作台。
4. `portfolio` 组合优化：底层模块存在，但没有服务层、API 与前端工作台。

这些能力彼此并非完全独立：

- 选股结果应能回灌回测。
- 诊股依赖行情、指标与研究上下文。
- 因子分析与组合优化共享行情、指标、导出与解释能力。
- 它们都应进入统一的登录保护、后端 API、前端工作台与文档边界。

## 目标澄清

本轮不是“把 placeholder 页面换成更好看的样式”，而是把四类能力补成真实可用的产品链路：

- 后端：有可调用、可测试、可文档化的真实 API。
- 前端：有可执行任务流、结果展示、失败态与导出路径。
- 能力协同：能与回测、行情、鉴权、OpenAPI 边界一致工作。

## 范围判断

### 纳入范围

- 选股服务正式迁入 `backend/app/services`，支持条件模板、自然语言选股转换、结果序列化与回测跳转。
- AI 诊股正式迁入后端服务，输出结构化诊断报告、指标摘要、风险提示与失败兜底。
- 因子分析与组合优化从“保留模块”升级为正式 API 能力。
- 前端新增对应四个工作台页，而不是继续沿用 skeleton brief。

### 不纳入范围

- 账户体系升级为多角色或多租户。
- 异步任务队列、数据库持久化和长期任务编排。
- 旧 Streamlit 页面复活。

## 关键分解

### 1. 选股链路

需要把当前页面里明确写出的几个缺口补齐：

- 东方财富指纹与自然语言选股请求链路迁入后端。
- 热门问句与筛选模板化。
- 选股结果支持排序、过滤、导出。
- 选股结果一键回灌回测。

### 2. AI 诊股链路

当前阻塞点也已经很清楚：

- 实时行情抓取。
- 技术指标计算。
- 提示词构建与模型调用。
- 报告渲染与失败兜底。

因此诊股不是单一接口，而是“行情 -> 指标 -> prompt/context -> 诊断报告”的复合链路。

### 3. 因子分析与组合优化链路

这两块已有 Python 模块与测试契约，但缺少：

- 服务层包装。
- 参数校验 schema。
- 正式 API。
- 前端结果可视化。
- 与市场/回测上下文的联动。

### 4. 共享能力

四类半成品能力需要共享以下通用层：

- 统一 market data / news / indicator 数据上下文。
- 统一错误模型与 degraded metadata。
- 统一导出格式与前端状态语言。
- 统一 OpenAPI 文档归口。

## 推荐实施顺序

```text
先打底层共享边界
  -> 因子分析 / 组合优化 API 化
  -> 选股能力正式迁移
  -> AI 诊股链路落地
  -> 前端四个工作台补齐闭环
```

原因：

- `factors` / `portfolio` 离“可用 API”最近，最适合作为第一批半成品转正能力。
- `screener` 依赖市场和回测联动，但产品价值高，适合作为第二批。
- `diagnosis` 外部模型与 prompt 复杂度最高，适合最后收口。

## 成功标准

- capability inventory 中这四类能力不再标记为 `skeleton`。
- backend 对应四组 API 有正式 schema、测试和 OpenAPI 描述。
- frontend 对应页面不再是 placeholder，而是可执行的工作台。
- 至少打通 2 条跨能力闭环：
  - 选股结果 -> 回测
  - 诊股 -> 指标/市场上下文报告

## 与其他提案关系

- 依赖 `adopt-akquant-backtesting-engine`：回测链路若被替换为 AKQuant，应作为选股回灌与研究联动的目标引擎。
- 依赖 `complete-backend-openapi-platform`：所有新 API 最终应纳入统一 OpenAPI。
- 依赖 `redesign-frontend-with-huashu-openapi`：前端工作台风格和交互在该提案中统一升级。
